# Shoemaker Lab Website — How It Works & How to Update It

*Last reviewed: July 2026*

## Overview

The site (`www.kevintshoemaker.net`) is a static site built with **Hugo** from source files stored in a **GitHub** repository. Updates are fully automated: pushing a change to GitHub triggers an automatic rebuild and redeploy — no manual file uploads needed.

## The Infrastructure Chain

```
Your computer (edit files)
      ↓ git push
GitHub repo: kevintshoemaker/Shoemaker-lab-website
      ↓ (webhook via GitHub App)
AWS Amplify (Ohio / us-east-2 region)
      ↓ runs `hugo` build command
      ↓ publishes /public folder
CloudFront (CDN — handles HTTPS/caching)
      ↓
Route 53 (DNS — points www.kevintshoemaker.net to CloudFront)
      ↓
Visitor's browser
```

**Key point:** Route 53 only handles DNS (translating the domain name to an address) and domain registration. It does not store any website files. The actual HTML/CSS/content is generated by Hugo and hosted via Amplify.

## How to Update the Website (Normal Workflow)

1. **Edit source files locally** — content in `content/`, templates in `layouts/`, site config in `config.toml`, etc.
2. **Preview locally** (optional but recommended): run `hugo server` in the repo folder and check `http://localhost:1313` in a browser.
3. **Commit and push** — via GitHub Desktop (recommended, especially for student contributors since it handles login without manual tokens):
   - Review changed files
   - Write a commit message
   - Commit to `main`
   - Push origin
4. **Amplify auto-builds** — within about a minute, the Amplify console (Ohio region) will show a new build running through Provision → Build → Deploy → Verify.
5. **Site updates live** automatically once the build succeeds — no manual CloudFront cache invalidation needed for normal content changes (Amplify handles this).

You never need to manually build Hugo and upload files — Amplify does the build server-side from your pushed source files.

## Known Gotchas (Learned During This Review)

- **Amplify is region-locked in the console.** If the Amplify app doesn't appear, check the region dropdown (top-right of AWS Console) — it's in **Ohio (us-east-2)**, not necessarily the default region shown.
- **GitHub Desktop or a Personal Access Token is required for pushes.** GitHub disabled plain username/password authentication in 2021. GitHub Desktop sidesteps this by handling OAuth login automatically.
- **`public/` is not currently in `.gitignore`.** This folder is Hugo's generated output and shouldn't need to be committed, since Amplify regenerates it on every build. Recommended addition to `.gitignore`:
  ```
  # Hugo build output
  /public/
  ```
  (If `public/` turns out to be tracked in the repo history, it can be untracked with `git rm -r --cached public` — worth doing as a cleanup pass, not urgent.)
- **No pinned Hugo version in Amplify.** The `amplify.yml` build spec doesn't specify `HUGO_VERSION`, so Amplify uses its own default. For consistency between local previews and deployed builds, consider adding an environment variable in Amplify (App settings → Environment variables): `HUGO_VERSION = 0.164.0` (or whatever your local version is at the time).
- **GitHub App migration completed (July 2026).** Amplify's connection to GitHub was migrated from the older OAuth method to the newer GitHub App method, as prompted by AWS. This is a one-time maintenance step and shouldn't need repeating.

## Quick Troubleshooting Checklist

If the site isn't updating after a push:
1. Check the Amplify console (Ohio region) for a failed build — click into the build for detailed logs.
2. Confirm the push went to the branch Amplify is actually watching (check the "Branches" tab in the Amplify app).
3. Confirm local Hugo version isn't producing errors that a different Amplify-side version wouldn't hit (check Amplify build logs for Hugo errors specifically).

If DNS/domain issues arise (site totally unreachable, not just outdated):
1. Check Route 53 → Hosted zones → confirm the `www` alias record still points to a valid CloudFront domain.
2. Check CloudFront → confirm the distribution still exists and is enabled.
